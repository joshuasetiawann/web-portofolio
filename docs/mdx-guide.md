# MDX Authoring Guide

How to write blog posts. Blog is the only content type compiled through MDX; everything else is typed data (see [`content-editing.md`](./content-editing.md)).

Posts live in `src/content/blog/*.mdx`. The **filename is the slug** — `src/content/blog/my-post.mdx` is served at `/blog/my-post`.

---

## How rendering works

```
src/content/blog/*.mdx
        │  Velite compiles + validates (velite.config.ts)
        ▼
   .velite/        (generated, gitignored)
        │  imported via the `#site/content` alias
        ▼
   src/lib/content.ts        (typed accessors — the only import boundary)
        │
        ▼
   @/components/mdx/mdx-content.tsx  -> MDXContent({ code })
        │  evaluates the compiled body against the registry
        ▼
   @/components/mdx/mdx-components.tsx  (themed element registry)
```

1. **Velite** reads each `.mdx` file, validates frontmatter against the schema in `velite.config.ts`, compiles the body to a function string (`s.mdx()`), and writes typed data to `.velite/`.
2. **`src/lib/content.ts`** wraps the generated collection and is the only module pages import from. Accessors: `getAllPosts()`, `getFeaturedPosts()`, `getPostBySlug()`, `getAllPostTags()`, `getRelatedPosts()`. Drafts are visible in dev and hidden in production; posts sort newest-first by `date`.
3. **`MDXContent`** (a server component in `mdx-content.tsx`) evaluates the compiled body string against the JSX runtime and renders it with the component registry.
4. **`mdx-components.tsx`** maps each markdown element to a themed, accessible React component. The MDX never reaches the runtime as raw user input — it is your own content, compiled at build time.

Compile content with:

```bash
pnpm content          # one-off compile (and frontmatter validation)
pnpm content:watch    # recompile on change while writing
```

`pnpm dev` and `pnpm build` run `velite` automatically, so a separate `pnpm content` is only needed for a manual compile or to surface validation errors early.

## Frontmatter

Frontmatter is YAML between `---` fences at the top of the file. Fields are defined and validated by the schema in `velite.config.ts`.

```mdx
---
title: Performance Budgets That Stick
description: How to set a performance budget the whole team will actually keep.
date: 2026-06-30
updated: 2026-06-30
tags: [Performance, Process]
draft: false
featured: true
cover: /static/covers/perf-budgets.png
---
```

| Field | Type | Required | Default | Notes |
| --- | --- | --- | --- | --- |
| `title` | string (≤120 chars) | **yes** | — | Post title |
| `description` | string (≤300 chars) | **yes** | — | Summary for listings, `<meta>`, and the auto-generated OG image |
| `date` | ISO date (`YYYY-MM-DD`) | **yes** | — | Publish date; sorts the feed |
| `updated` | ISO date | no | — | Last-updated date |
| `tags` | string array | no | `[]` | Powers tag lists and related-post matching |
| `draft` | boolean | no | `false` | `true` hides the post in production (still visible in dev) |
| `featured` | boolean | no | `false` | Surface in featured rails |
| `cover` | string | no | — | Path to a cover image |

**Derived automatically — do not author these:**

- `slug` — from the filename.
- `url` — `/blog/<slug>`.
- `readingTime` — minutes, computed from word count (minimum 1).
- `metadata` / `toc` — reading-time data and a table of contents, generated by Velite (`s.metadata()`, `s.toc()`).
- `body` — the compiled MDX (`s.mdx()`).

Validation is strict: exceed a max length or use a malformed date and `pnpm content` (and the build) fails with a precise error.

## Supported elements

The registry in `mdx-components.tsx` styles standard Markdown — write normal Markdown and it renders themed. Mapped elements: `h1`–`h4`, `p`, `a`, `ul`, `ol`, `li`, `blockquote`, `hr`, `strong`, `em`, `code` (inline), `pre` (fenced blocks), `img`.

### Headings

```mdx
## Section heading      (h2 — gets a bottom border)
### Subsection          (h3)
```

Headings carry `scroll-m-20` so in-page anchor jumps land cleanly. Use `##` and below for body structure; `#` (h1) is reserved for the post title rendered by the page layout.

### Text, lists, emphasis

Standard Markdown: paragraphs, `**bold**`, `_italic_`, bulleted (`-`) and numbered (`1.`) lists, and `---` for a horizontal rule — all themed to the design tokens.

### Links

`[label](href)` resolves by destination automatically:

- **External** (`http://`, `https://`, `//`) -> rendered via the `ExternalLink` component (opens safely in a new tab).
- **`mailto:` / `tel:`** -> a plain anchor with the same link styling.
- **Internal** (e.g. `/projects/aurora`) -> Next.js `<Link>` for client-side navigation.

You do not choose the link type — write the URL and the registry picks the right component.

### Callouts (blockquotes)

There is no custom `<Callout>` component. Use a Markdown blockquote for asides and callouts; it renders as a left-bordered, muted, italic block:

```mdx
> A great poster image is the cheapest performance win in WebGL.
```

### Code

**Inline:** wrap in backticks — `` `--primary` `` renders as a bordered inline chip.

**Fenced blocks:** use triple backticks with a language for syntax labeling. Fenced blocks render through the themed `CodeBlock`, which adds a **copy button** (appears on hover/focus) so readers can copy the snippet:

````mdx
```css
:root {
  --primary: #3d5be0;
}
```
````

The language tag (`css`, `ts`, `tsx`, `bash`, …) is read from the fence info string. A fenced block always becomes a `CodeBlock` with a copy button; filenames are not derivable from a plain Markdown fence, so blocks render without a filename header.

### Images

`![alt text](src)` renders a lazy-loaded, rounded, bordered `<img>`. **Always provide alt text** for accessibility. Place image assets under `public/` (served from the site root) or reference a processed asset under `/static/`.

```mdx
![The hero particle field at rest](/static/posts/helios-hero.png)
```

> Note: MDX images use a plain `<img>` (intrinsic dimensions are unknown at author time), not `next/image`. Optimize large images before adding them.

## Checklist for a new post

1. Create `src/content/blog/<slug>.mdx`.
2. Fill required frontmatter: `title`, `description`, `date`.
3. Write the body in Markdown/MDX using the elements above.
4. Run `pnpm content` (or rely on `pnpm dev`) and confirm it compiles without validation errors.
5. Verify `/blog` lists it and `/blog/<slug>` renders correctly.
6. Set `draft: false` (and optionally `featured: true`) when ready to publish.

See [`content-editing.md`](./content-editing.md) for all non-blog content and [`CONTENT-CHECKLIST.md`](./CONTENT-CHECKLIST.md) for the full intake list.
